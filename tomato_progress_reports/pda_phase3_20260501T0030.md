## DEFECT-45 — Ambiguity — Section 8.3 + Section 27 Fix-16: scope of "request confirmation" for intra-spec conflicts

**Class:** Ambiguity
**Severity:** HIGH
**Location:** Section 8.3 (section15-encoder agent body) + Section 27 Fix-16

**Quoted defect text (Section 27 Fix-16):**

> "When multiple conflicting values exist for a single scenario field at different spec locations, treat the scenario body text as authoritative. Report the conflict in `tomato_blockers.md` with all locations and values quoted, and **request confirmation before encoding that scenario.**"

**What happened in Phase 3:**
The encoder found 7 subsection-vs-body conflicts (SPEC-INT-002 cases: S3B.4, S3C.8, S3C.9, S3C.12, S3D.5, S3D.7, SB.11). It applied "scenario body wins" autonomously across all 7, encoded the tests, and documented the decisions in batch in `spec_changelog.md` after encoding. No per-conflict BLK entry was filed before encoding. No per-conflict "request confirmation" pause occurred.

**Why this is a defect:**
Fix-16 says "request confirmation before encoding that scenario." The encoder instead applied a generalization (the body-wins principle from the established S1.1 example) without stopping for per-case confirmation. The instruction is ambiguous on a critical question: does "request confirmation" mean (a) per-conflict, stop and ask, or (b) apply the established rule and document in batch? The encoder chose (b). A different agent reading the same instruction could choose (a), creating 6 separate BLK blockers before Phase 3 could complete. The instruction does not distinguish between "a novel conflict class not yet resolved" and "additional instances of an already-resolved conflict class."

**Evidence:** `tomato_log.md` [2026-04-30 17:15] records the encoder's 7 autonomous decisions. SPEC-INT-002 was written post-encoding in the cleanup pass [2026-05-01 00:30]. No `tomato_blockers.md` entries for S3B.4 through SB.11 exist.

**Phase 4 blocking:** No — encoder phase is closed. Applies to future Phase 3 re-runs only.

**Suggested fix:** In Section 8.3 and Section 27 Fix-16, add a pattern-application exception:

> "If the conflict matches an already-resolved conflict class (same type, same resolution rule, documented in a prior SPEC-INT-NNN entry in `spec_changelog.md`), the encoder may apply the rule autonomously and document each case with an inline NOTE comment and in a batch SPEC-INT-NNN entry. Per-case BLK filing and per-case confirmation are NOT required for pattern re-applications. If the conflict is a novel class with no prior SPEC-INT entry, stop, file a BLK entry, and request confirmation. The SPEC-INT-001 (S1.1 case) established body-wins for intra-Section-15 conflicts; subsequent subsection-heading-vs-body conflicts are pattern re-applications of the same class."

## DEFECT-46 — Missing instruction — Section 11.4b: SPEC-INT-NNN format not defined (upgraded from Defect-43 LOW to HIGH)

**Class:** Missing instruction
**Severity:** HIGH (Defect-43 was LOW; severity upgraded because SPEC-INT-002 is now a load-bearing artifact for Phase 4)
**Location:** Section 11.4b (spec_changelog.md format block)

**Quoted defect text (Section 11.4b) — only format defined:**

```
## SPEC-CHG-NNN [YYYY-MM-DD] <title>
- Spec section affected: ...
- Original text: <verbatim>
- New text: <verbatim>
- Reason: ...
- User approval (verbatim quote): ...
- Tasks affected: T-NNN, T-MMM (require re-audit)
```

**The gap:** Section 11.4b defines only `SPEC-CHG-NNN` (text-change entries). It has no definition for `SPEC-INT-NNN` (interpretation entries that resolve contradictions without modifying spec text). Both SPEC-INT-001 and SPEC-INT-002 are now on disk and load-bearing: they establish which scenario input vectors and which tier outcomes the 135 tests use. Phase 4 T-IMPL-5a must know these interpretations are authoritative.

**Why severity is HIGH now:** A fresh-session Phase 4 implementer reading only the master prompt would see SPEC-INT-NNN entries in `spec_changelog.md` and have no protocol anchor for them. The format is self-described in both entries as "consistent with what Defect-23 will codify" — meaning the format is pending normalization. If the implementer treats SPEC-INT entries as non-binding (because they are not listed in master prompt Section 11.4b), it could re-investigate S1.1 or the 7 SPEC-INT-002 scenarios from scratch, potentially reaching different conclusions.

**Phase 4 blocking:** Not hard-blocking; the implementer can read spec_changelog.md directly. But the lack of normative status creates interpretation risk.

**Suggested fix:** Add to Section 11.4b, after the SPEC-CHG-NNN block:

> "**SPEC-INT-NNN entries** are used when a contradiction is resolved by interpretation (one location is declared authoritative) rather than by modifying spec text. Format:
>
> ```
> ## SPEC-INT-NNN [YYYY-MM-DD] <title>
> - Note: This is an INTERPRETATION entry, not a text-change entry.
> - Spec sections affected: ...
> - Conflicting locations: (quote both)
> - Authoritative location: (quote and explain why it wins)
> - Non-authoritative location: (quote and explain why it loses)
> - Resolution rule applied: (reference Fix-NN or DEC-NNN)
> - Encoder/implementer enforcement status: (applied / pending)
> - Tasks affected: (list tasks; note if no re-audit needed because encoder already applied)
> - User approval (verbatim quote): ...
> - Status: RESOLVED via interpretation. Spec text unchanged.
> ```
>
> SPEC-INT-NNN entries are BINDING for all downstream subagents. An implementer must not re-investigate a conflict already resolved by a SPEC-INT-NNN entry unless a new BLK is filed arguing the resolution is wrong."

## DEFECT-47 — Missing instruction — Sections 11 / 23: sacred manifest evolution protocol absent

**Class:** Missing instruction
**Severity:** HIGH
**Location:** Section 23.2 (operations requiring explicit user approval) + Section 2 (sacred files) + Section 8.5 (sacred-guardian)
**Log entry that revealed the gap:** `tomato_log.md` [2026-05-01 00:30], DEC-019

**What happened:**
DEC-019 introduced three schema-level changes to `.claude/sacred_manifest.json`: (1) added `log_exclusions: ["*.log", "*.log.*"]` to the `scripts/apin/` entry, (2) updated the `directory_hash_algorithm_canonical` pseudocode to honor per-entry `log_exclusions`, and (3) added a `rebaseline_history` array field. The manifest now has a structurally non-uniform schema (one entry has fields others do not).

**The defect:**
Section 23.2 says "Modifying the sacred manifest (per Section 8.5)" requires user approval. Section 8.5 (sacred-guardian) describes only hash verification, not what constitutes a valid manifest schema, which fields are permitted, how new fields are authorized, or what "rebaseline" means procedurally. There is no protocol for:
- What fields are allowed in a manifest entry
- How to add a `log_exclusions` field (schema evolution vs hash update)
- What "rebaseline" means and when it is authorized
- Whether the canonical hash algorithm pseudocode is itself sacred (it lives in the manifest; changing it changes what "drift" means for that directory)
- What DEC entry is required for a schema evolution vs a simple hash update

A fresh-session agent reading the master prompt cannot determine from Section 23.2 alone whether DEC-019 required a SPEC-CHG entry, a DEC entry, or a blocker. The user had to specify the exact field name, pattern, and algorithm update from scratch because no template existed.

**Phase 4 blocking:** No. DEC-019 is complete and verified. This defect matters if manifest evolution recurs (e.g., when the LoRA file is copied to sandbox in Phase A.3 and becomes sacred — that would be a new-entry addition).

**Suggested fix:** Add Section 11.7 "Sacred manifest governance":

> "The sacred manifest at `.claude/sacred_manifest.json` is maintained (not sacred itself) but governs sacred file protection. Three categories of manifest change have different approval requirements:
>
> - **Hash-only update** (a tracked file was legitimately modified, e.g., T-EARLY-MP edits `tomato_master_prompt.md`): update the sha256 for the affected entry. Log in `tomato_log.md`. No DEC needed. Run sacred-guardian to confirm the new hash is correct.
>
> - **New entry** (a new file becomes sacred, e.g., LoRA copy in Phase A.3): add the entry with all required fields (`type`, `sha256`, `size_bytes` for files; `type`, `sha256`, `file_count`, `algorithm` for directories; `spec_reason`). Log as DEC entry with user approval quote.
>
> - **Schema evolution** (adding new fields to an existing entry, changing the canonical hash algorithm pseudocode, adding `rebaseline_history`): requires explicit user approval specifying the exact field names and values. Log as DEC entry with user approval quote. Update the `directory_hash_algorithm_canonical` pseudocode if the hashing behavior changes. Run main-thread independent canonical verification after.
>
> The `directory_hash_algorithm_canonical` pseudocode in the manifest is the normative algorithm for the `sacred-guardian` to follow. Any change to it requires schema-evolution approval."

## DEFECT-48 — Missing instruction — Section 4: no Phase 4 entry preconditions block

**Class:** Missing instruction
**Severity:** MEDIUM
**Location:** Section 4, between Phase 3 exit and Phase 4 entry

**Quoted defect text (Section 4 Phase 3 exit — the only gate):**

> "Exit: 135 tests encoded; all 135 fail with expected failure modes; test infrastructure in place; Section 15 protection hook installed and verified. STOP and report. Wait for user approval before Phase 4."

**The gap:** The master prompt has no Phase 4 entry preconditions block equivalent to the plan's "Phase 3 Entry Preconditions" table (which was patched twice: once to add spec_changelog.md gate, once via DEC-017 to correct the T-IMPL-5a/5b inversion). Specific conditions that are silently assumed but not stated:

1. `.claude/import_contract.md` must exist and specify `assign_tier()` signature and all symbols the 135 tests import. The implementer must honor these paths exactly. No master prompt instruction says "verify import_contract.md before Phase 4."
2. The Section 15 pre-commit hook must be verified ACTIVE in the new session. Phase 3 step 6 installs it, but if Phase 3 ran in a session that ended, the new session has no explicit instruction to re-verify the hook before dispatching the implementer.
3. SPEC-INT-NNN entries (SPEC-INT-001, SPEC-INT-002) must be acknowledged before the implementer starts T-IMPL-5a. No master prompt instruction says the implementer reads spec_changelog.md at Phase 4 entry.
4. `tomato_blockers.md` must have no open entries. The master prompt does say blockers must be resolved before continuing; but this is scattered across several sections rather than collected into a Phase 4 gate.

**Phase 4 blocking:** Soft — user will approve Phase 4 entry and can confirm these conditions verbally. But without a checklist, a fresh-session agent could skip verification.

**Suggested fix:** Add after Phase 3 exit in Section 4:

> "**Phase 4 entry conditions (all must be TRUE before dispatching implementer subagent):**
> 1. Phase 3 user-approved (phase exit gate READY verdict received).
> 2. `.claude/import_contract.md` exists and specifies `assign_tier()` signature and all symbols imported by the 135 Section 15 tests.
> 3. Section 15 pre-commit hook is active: verify `ls .git/hooks/pre-commit` returns the file and `cat .git/hooks/pre-commit` shows the guard pattern `test_section15_*.py`.
> 4. `spec_changelog.md` contains at least SPEC-INT-001; any additional SPEC-INT entries from Phase 3 are reviewed by the implementer before starting tier-assignment work.
> 5. `tomato_blockers.md` has no OPEN entries."

## DEFECT-49 — Contradiction — Section 27 Fix-16 "request confirmation" vs SPEC-INT-002 accepted batch practice

**Class:** Contradiction
**Severity:** MEDIUM
**Location:** Section 27 Fix-16 (the instruction) vs the user-approved outcome in SPEC-INT-002 + DEC-019

**Quoted defect text (Section 27 Fix-16):**

> "Report the conflict in `tomato_blockers.md` with all locations and values quoted, and **request confirmation before encoding that scenario.**"

**Quoted user approval (SPEC-INT-002, DEC-019 user message):**

> "Body-wins decisions: single SPEC-INT-002 entry covering all 7 cases. Same root cause, same resolution rule, batch documentation."

The user accepted a batch-documentation approach that explicitly did NOT require 7 separate BLK entries filed before encoding. Fix-16's instruction ("request confirmation before encoding that scenario") is now contradicted by an accepted precedent that batch-documenting pattern re-applications without per-case confirmation is correct behavior.

**Distinction from Defect-45:** Defect-45 is an ambiguity (the instruction could be read either way). Defect-49 is a contradiction: one reading of Fix-16 is now definitively wrong based on the accepted precedent, but Fix-16 still says it. Fix-16 needs to be updated to match the accepted practice.

**Phase 4 blocking:** No. Affects future Phase 3 re-runs only.

**Suggested fix:** Same as Defect-45 — update Fix-16 to add the pattern-application exception clause. Resolving Defect-45 also closes Defect-49.

## DEFECT-50 — Missing instruction — Section 8.3: file organization list incomplete relative to spec Section 15 structure

**Class:** Missing instruction
**Severity:** LOW
**Location:** Section 8.3 (section15-encoder agent body, "File organization for the 135 tests" block)

**Quoted defect text (Section 8.3):**

```
File organization for the 135 tests. Group by tier:
- test_section15_tier1.py - Tier 1 scenarios
- test_section15_tier2.py - Tier 2 scenarios
- test_section15_tier3a.py - Tier 3A scenarios
- test_section15_tier3b.py - Tier 3B scenarios
- test_section15_tier4a.py - Tier 4A scenarios
- test_section15_tier4b.py - Tier 4B scenarios
- test_section15_tier5.py - Tier 5 scenarios
- test_section15_special.py - Special / cross-tier scenarios
```

**What the encoder produced (13 files):**
tier1, tier2, tier3a, tier3b, tier3c, tier3d, tier4a, tier4b, tier5, boundary, underpowered, disagreement, tta.

The 5 extra files (tier3c, tier3d, boundary, underpowered, disagreement, tta) map to real Section 15 subsections not enumerated in the master prompt list. The list is not marked "minimum" or "extend as needed." A strict reading — "Group by tier: [8 files listed]" — could cause a future encoder to force all 135 scenarios into 8 files, misattributing Tier 3C/3D scenarios to "special" and losing subsection organization.

**Evidence:** `tomato_log.md` [2026-04-30 17:15] lists 13 files explicitly; master prompt list has 8.

**Phase 4 blocking:** No. Tests are on disk. Defect applies to future Phase 3 re-runs.

**Suggested fix:** Replace the fixed list in Section 8.3 with:

> "Group by tier following Section 15's actual subsection structure. Starting point (extend as needed based on Section 15's actual subsections):
> - test_section15_tier1.py, tier2.py, tier3a.py, tier3b.py, tier3c.py, tier3d.py, tier4a.py, tier4b.py, tier5.py
> - test_section15_boundary.py — boundary and edge-case scenarios
> - Additional files for signal-disagreement, TTA, underpowered, and other named subsections in Section 15
>
> **Do not force scenarios from unlisted subsections into `test_section15_special.py`.** Create one file per Section 15 subsection that contains 5 or more scenarios. The final file count depends on spec Section 15's subsection structure; 8 files is a minimum, not a cap."

## DEFECT-51 — Missing instruction — Section 8.3: test function naming convention is example-only, not normative

**Class:** Missing instruction
**Severity:** LOW
**Location:** Section 8.3 (section15-encoder agent body, naming instruction)

**Quoted defect text (Section 8.3):**

> "Test function or parametrize ID matches scenario ID (e.g., `def test_S1_1`, `pytest.param(..., id='S1_2')`)"

**What happened:** 9 of 13 test files use short form `def test_S*_*()`. 4 files use long form `def test_scenario_S*_*()`. The master prompt example shows `def test_S1_1` (short form), but since the instruction says "e.g." rather than "must be exactly", the encoder treated it as illustrative. The result is a mixed naming convention across 13 files (4 vs 9 file split), documented as Defect-45-process-note in `tomato_log.md` and queued as Fix-45 in `tomato_plan.md` for T-EARLY-MP cleanup.

**Why it is a defect in the master prompt:** The convention is given by example with "e.g." which is non-normative. If this is intended to be a binding naming rule (as it appears to be — the T-EARLY-MP queue item says "rename short-form functions to verbose form"), the master prompt must state it normatively. If "e.g." is intentional flexibility, the T-EARLY-MP cleanup item is unnecessary work.

**Phase 4 blocking:** No. Both naming forms are pytest-discoverable; all 135 tests run. T-EARLY-MP Fix-45 can normalize later.

**Suggested fix:** Replace the "e.g." form with a normative statement:

> "Test function naming: use `def test_S<tier_id>_<number>()` for individual functions (e.g., `def test_S1_1()`). For parametrized groups use `pytest.param(..., id='S<tier_id>_<number>')`. The prefix `test_scenario_` is NOT used. Apply this convention uniformly across all 13 files. Mixed naming in the same file set is a defect."
